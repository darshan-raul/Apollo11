## Critical Architecture Decisions (Do Not Revert)

1. **Frontend is NGINX-only on port 3000.** Five probe paths all return 200 unconditionally. No supervisord, no separate Go probe server. The "supervisord + port 8085" decision in older notes is **wrong** — the actual code (Stage 4+) is NGINX-only.

2. **`vite.config.js` reads `VITE_*` from `process.env`.** The `define` block at the bottom of the file is now `process.env.VITE_FOO || 'fallback'` — never hardcoded URLs. This was the bug that prompted `d53f414` and `b968cb9`. If a future session adds a new stage with a different frontend URL pattern, **make sure the `define` block reads from `process.env`**, or the build-arg VITE_* won't take effect.

3. **DATABASE_URL `***` pattern** in docker-compose.yml is intentional — `***` is a literal string in YAML, password resolves correctly in container env (not a bug).

4. **Graceful SIGTERM**: Go services use `context.WithTimeout(30s)`. Python Identity uses `signal.signal()` (just logs, then uvicorn's built-in drain does the actual work). NGINX receives SIGTERM and exits within ~1s.

5. **Docs repo is Docusaurus** (not MkDocs). Build via `npm run build`, config in `docusaurus.config.ts`, custom CSS in `src/css/custom.css`, pages in `src/pages/`.

6. **No npm on host** — frontend builds happen inside Docker multi-stage (node builds → nginx serves).

7. **Stage 2 access stack** (Traefik or Envoy + MetalLB) is **carried over verbatim** into stages 3-7 manifests. Stage 5's `set5-envoy-gateway/` is the canonical reference for this layout. Stage 4+ code (probes, graceful shutdown) and stage 6+ code (OTEL) are layered on top.

8. **Envoy Gateway v1.5.0** is the version pinned in stages 3+ and Stage 2 set 5. The version-sweep in `stages/stage2/NOTES.md` confirmed v1.2.4 → v1.5.0 all work; v1.5.0 was chosen for having been already validated in-session.

9. **Helm chart bundles the access stack** (Stage 5). One `helm install` provisions the apps + StatefulSets + Envoy Gateway + MetalLB. Kustomize path is a plain-manifest subset (apps only) for dev-friendly iteration.

After the Stage 2 restage (5 sets, v1.5.0) + Stage 3 audit (vite.config.js

## Gaps found and fixed

### 1. `vite.config.js` hardcoded URLs in stages 4/5/6 (CRITICAL — would fail at runtime)

The exact bug fixed in `d53f414` and `b968cb9` was **not propagated** to
stages 4, 5, or 6. The `define` block had hardcoded `localhost:8080-8083`
URLs that silently overrode the `--build-arg VITE_*=...` build args.

**Impact:** the browser would have called `http://localhost:8080` and
failed to reach any backend through Envoy Gateway. The frontend pod
would still be "Ready" (the NGINX probe paths all return 200), but every
user action would have 404/failed to connect.

**Fix:** synced `stages/stage3/code/frontend/vite.config.js` (the fixed
version with `process.env.VITE_* || 'fallback'`) to stages 4, 5, 6.
All 4 vite.config.js files now have identical md5.

### 2. Envoy Gateway v1.2.4 vs v1.5.0 drift in stages 4/5/6 (CONSISTENCY)

Stages 4/5/6 still bundled the v1.2.4 install (2.5MB) but Stage 2 set 5
and Stage 3 had been bumped to v1.5.0 (2.9MB) in `b968cb9`.

**Impact:** the v1.2.4 install would still **work** with stages 4/5/6's
apply.sh (the apply scripts are written for the v1.5.0 flow with no
Service patch step), but the canonical reference is v1.5.0 — and the
v1.5.0 bundle was already sitting in `stages/stage3/k8s/gateway/`.

**Fix:** copied the v1.5.0 install.yaml from stage 3 to:
- `stages/stage4/k8s/gateway/00-envoy-gateway-install.yaml`
